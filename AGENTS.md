# Agents Guide — markdown

## What this project is

`@tabnas/markdown` is a **record/field grammar plugin** for the
[`tabnas`](https://github.com/tabnas/parser) parsing engine. It turns
delimited record text into arrays of objects (or arrays of arrays), with
headers, RFC 4180 `""`-escaped quoting, custom field/record separators,
streaming, and a strict / non-strict mode. The entry rule is `markdown`
and a record is a comma-separated (`#CA`) list of fields, one record per
line (`#LN`).

The repository was created from the
[`@tabnas/csv`](https://github.com/tabnas/csv) template and still shares
its CSV-shaped record/field model — the grammar rules are
`markdown` / `newline` / `record` / `text`, the fixtures are `.csv` →
`.json` pairs, and the error codes are `markdown_extra_field` /
`markdown_missing_field`. Don't be misled by the package name: the plugin
parses delimited rows, not GitHub-Flavored Markdown.

It is **not** standalone: it layers on the
[`@tabnas/jsonic`](https://github.com/tabnas/jsonic) base grammar (the
relaxed-JSON `val`/`map`/`list`/`pair`/`elem` rules) and reuses the
engine's lexer, comment handling, and streaming hooks. In **strict mode**
(the default, `strict: true`) jsonic value parsing is switched off and
every field is raw text; in **non-strict mode** a field body can hold
embedded jsonic (`[1,2]`, `{x:1}`), so the plugin preserves jsonic's
default `val`/`elem`/`list` alternatives.

There are two implementations that must behave identically — TypeScript
(canonical) and a Go port.

## The tabnas engine dependency

Both runtimes depend on the unpublished `@tabnas` siblings via a
**sibling checkout** (the standard tabnas dev model until the packages
publish tagged releases):

- TypeScript: `ts/package.json` declares `@tabnas/parser`
  (`peerDependency` `">=2"`) and `@tabnas/jsonic` (`peerDependency`
  `file:../../jsonic/ts`). Both are mirrored as
  `file:../../parser/ts` / `file:../../jsonic/ts` devDependencies for
  local builds (npm >=7 / Node >=24 auto-installs peers; `engines.node`
  is `">=24"`). `@tabnas/debug` and `@tabnas/railroad` are dev-only
  `file:` devDependencies — debug for the `debug-model.test.ts`
  composition test, railroad to regenerate `ts/doc/grammar.{svg,txt}`.
- Go: `go/go.mod` requires `github.com/tabnas/jsonic/go` with
  `replace github.com/tabnas/jsonic/go => ../../jsonic/go`. That is the
  module's **only** tabnas dependency — the Go jsonic package re-exports
  the engine types (`jsonic.Jsonic`, `jsonic.Options`, `jsonic.Rule`,
  `jsonic.LexOptions`, …), so `markdown.go` imports `jsonic`, not
  `parser`, directly.

Clone the siblings (`parser`, `jsonic`, plus `debug`/`railroad` for the
optional tests and diagram) next to this repo and build their TS first.
CI does this for you (see below).

## Authority and alignment rules

**TypeScript is canonical. Go is a port of it.** When you change
behaviour:

1. Change `ts/src/markdown.ts` first (or
   `ts/markdown-grammar.jsonic` for grammar changes — see the embed
   section below).
2. Port the same change to `go/markdown.go`.
3. Add/extend the shared fixture(s) in `test/fixtures/` + `manifest.json`
   so both runtimes assert the new behaviour. The fixtures are the parity
   contract; both suites resolve the shared `test/fixtures/` dir (TS:
   `ts/test/markdown.test.ts` `fixturesDir` = `../../test/fixtures`; Go:
   `go/markdown_test.go` `fixturesDir()` = `../test/fixtures`).
4. Mirror any new unit cases across `ts/test/markdown.test.ts` and
   `go/markdown_test.go` — the two unit suites should cover the same
   ground.
5. Run both suites and confirm green.

Do not let the Go behaviour drift from TS. If Go genuinely cannot match
because of a `jsonic/go` limitation, document the gap here and in the
relevant `doc/*.md` rather than silently diverging.

## The grammar is embedded — never hand-edit the embedded block

`ts/markdown-grammar.jsonic` is embedded verbatim into **both**
`ts/src/markdown.ts` and `go/markdown.go`, between these markers:

```
// --- BEGIN EMBEDDED markdown-grammar.jsonic ---
...
// --- END EMBEDDED markdown-grammar.jsonic ---
```

Edit `ts/markdown-grammar.jsonic`, then run the embed step. Never edit the
text between the markers by hand — it will be overwritten. (The grammar
may not contain backticks; `embed-grammar.js` aborts if it does, since the
Go side uses a raw string.)

```bash
cd ts && node embed-grammar.js   # writes into ts/src/markdown.ts AND go/markdown.go
```

`npm run build` runs the embed step first (`node embed-grammar.js && tsc
--build src test`), so a normal TS build keeps both files in sync.

The `markdown`, `newline`, `record`, and `text` rules live in the grammar
file. The `list`, `elem`, and `val` rules are configured **in code
instead** (`tn.rule(...)` in TS / `j.Rule(...)` in Go), because
non-strict mode must preserve jsonic's default alternatives for those
rules so embedded JSON values keep working.

## Architecture notes

- **Strict mode (default).** Jsonic value parsing is disabled; every
  field is raw text, and the plugin's custom RFC 4180 string matcher
  (`buildMarkdownStringMatcher`) handles `""`-escaped quotes (gated on
  `options.string.markdown !== false`). The JSON structural tokens
  (`#OB`, `#CB`, `#OS`, `#CS`, `#CL`) are switched off, and
  `rule.exclude: 'jsonic,imp'` drops the relaxed alternatives.
- **Non-strict mode** (`strict: false`). Field bodies parse as jsonic, so
  a cell can hold `[1,2]`, `{x:1}`, or a backslash-escaped string.
  Non-strict also defaults `trim`, `comment`, `number`, and `value` to on,
  and excludes only `imp`.
- `#LN` (line end) is removed from the IGNORE token set so row breaks are
  significant; in strict mode `#SP` is also removed so in-field whitespace
  survives.
- Per-instance options are applied **after** the grammar is installed
  via `tn.grammar(grammarDef)`, because the grammar's internal
  `SetOptions` would otherwise clobber the `tokenSet`/`fixed` token
  overrides.
- The grammar text is parsed by a **separate** engine instance
  (`new Tabnas().use(jsonic).parse(grammarText)` in TS), then `grammarDef.ref`
  is set to the `refs` map and the spec is handed to `tn.grammar(...)`.
  `@`-prefixed names in the grammar resolve against `refs`: state actions
  are auto-wired by the `@rulename-{bo,ao,bc,ac}` convention, while alt
  actions, condition refs, and funcref names (`@text-follows`,
  `@not-record-empty`, `@record-close-next`, …) are referenced explicitly.
- Token legends: the plugin registers human descriptions for markdown
  tokens via `config.modify` → `cfg.tokenDesc` (`'markdown-tokendesc'`),
  which `@tabnas/railroad` reads off the live config when drawing the
  diagram legend.
- `header`, `object`, `field.exact`, `field.names`, `field.nonameprefix`,
  `field.empty`, and `record.empty` shape the `@record-bc` action;
  `field.exact` raises `markdown_extra_field` / `markdown_missing_field`
  via `ctx.t0.bad(code)` when a row's field count differs from the header.
  Streaming (`stream`) emits `start` / `record` / `end` / `error` and
  suppresses the in-memory result.

## Build & test

TypeScript (from `ts/`):

```bash
npm install            # auto-installs peers; resolves file: siblings
npm run build          # embeds grammar, then tsc --build src test
npm test               # node --test over dist-test/*.test.js
```

Go (from `go/`):

```bash
go build ./...
go test -v ./...       # unit tests + shared fixtures
```

The repo root [`Makefile`](Makefile) (adapted from voxgig/util) wraps
both halves: `make build|test|clean` run the TS and Go sides, and
`make publish-go V=x.y.z` injects `V` into the `const Version` in
`go/markdown.go` (currently `0.1.1`), commits, and tags `go/vX.Y.Z`.
`make publish-ts` publishes the TS package at its `package.json` version
(currently `0.1.1`). There is also a TS-side [`ts/Makefile`](ts/Makefile)
with the same target shape. The `Version` const is kept in sync across
runtimes by `make publish-go`.

Local builds resolve the unpublished siblings via the repo-set `go.work`
+ node_modules symlinks created by `admin/scripts/link.sh`; there is no
checked-in `go.work` in this repo.

## Optional tests (@tabnas/debug)

`ts/test/debug-model.test.ts` composes the markdown plugin with the
external `@tabnas/debug` plugin and asserts the structured grammar model:
the rule set (`markdown`, `newline`, `record`, `text` plus the shared
jsonic `val`/`map`/`list`/`pair`/`elem`), `m.config.start === 'markdown'`
(note `config.start`, not `m.start`), that a plugin named `Markdown` is in
`m.plugins`, and that `markdown` pushes `newline`/`record` while `record`
pushes `list`. It resolves debug dynamically and **skips** unless
`@tabnas/debug` is installed (it is a `file:` devDependency, so `npm test`
runs it) or `TABNAS_DEBUG_PATH` points at a built sibling.

`ts/test/doc-examples.test.ts` extracts fenced `js`/`javascript` blocks
containing `// =>` assertions and runs them. It globs candidate doc
locations (`README.md`, `ts/README.md`, `go/README.md`, `ts/doc/`, `doc/`,
`docs/`) and skips any that are absent; in this repo that resolves to
`README.md`, `ts/README.md`, and the `ts/doc/*.md` files. Keep doc
examples correct.

## CI

`.github/workflows/build.yml` has two jobs, neither publishing to npm:

- **build** (Ubuntu/Windows/macOS, Node 24): sets
  `git config --global core.autocrlf false` (CRLF corrupts the `.csv` /
  `.json` fixtures, which are marked `binary` in `.gitattributes`),
  git-clones the tabnas closure (`parser debug json abnf railroad jsonic`)
  as siblings, `npm i && npm run build --if-present` each in topo order,
  then `npm test` in `markdown/ts`. Because `@tabnas/debug` is a
  devDependency, the composition test runs as part of `npm test`.
- **build-go** (Ubuntu/macOS, Go 1.24): clones the same siblings, mirrors
  `admin/scripts/link.sh` by creating `vendor/` symlinks for any
  `../vendor/` replaces and a `go work` over every non-vendor-replaced
  module, then `go build ./...` / `go test -v ./...` in `markdown/go`.
